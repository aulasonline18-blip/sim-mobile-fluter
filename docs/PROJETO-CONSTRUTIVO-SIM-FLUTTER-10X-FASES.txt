======================================================================
0. COMO USAR ESTE DOCUMENTO
======================================================================

Este documento divide a planta SIM Flutter 10x em fases construtivas.
Ele deve ser seguido em sequencia. Cada fase tem:

- objetivo;
- por que vem nessa ordem;
- orgaos arquiteturais envolvidos;
- entregaveis;
- passos construtivos;
- testes obrigatorios;
- criterio de aceite;
- proibicoes;
- saida esperada.

Regra principal:

Nao avance para a fase seguinte se a fase atual nao estiver testada,
documentada, commitada e empurrada para `main`.

Regra de arquitetura:

Toda correcao ou melhoria deve entrar no orgao correto da Planta-Mae.
UI renderiza. Estado fica no StudentLearningState/Store. Pedagogia fica no
motor. IA fica atras de contrato. Sync fica no orgao de sync. Midia fica no
orgao de midia.

Regra de produto:

O objetivo nao e "ficar bonito". O objetivo e fazer o aluno aprender mais,
com menos atrito, com mais continuidade e com prova.

======================================================================
FASE 0 - BASELINE, SEGURANCA E CANTEIRO DE OBRA
======================================================================

Objetivo:

Criar o ponto de partida confiavel. Antes de melhorar, provar que sabemos
voltar, medir, testar e nao quebrar o que ja existe.

Por que vem primeiro:

Sem baseline, qualquer melhoria vira aposta. Esta fase cria a linha de base
para comparar antes/depois e protege o fluxo vital.

Orgaos envolvidos:

- repo Flutter;
- repo API se necessario;
- docs;
- CI/local scripts;
- testes existentes;
- APK production.

Entregaveis:

1. Documento `docs/10x-baseline-report.md`.
2. Lista dos testes que definem "nao quebrar".
3. APK baseline com SHA256.
4. Lista de dispositivos reais de teste.
5. Checklist de fluxo vital.

Passos construtivos:

1. Rodar no Flutter:
   - `git status --short`
   - `git branch --show-current`
   - `git pull --rebase origin main`
   - `flutter analyze`
   - `flutter test`
   - `flutter build apk --release --dart-define=FLUTTER_APP_MODE=production`

2. Se API for tocada nesta fase, rodar:
   - `npm test`

3. Gerar APK e SHA256.

4. Publicar APK no link padrao:
   - `http://167.179.109.137:3000/downloads/sim-production-latest.apk`

5. Registrar:
   - commit atual;
   - APK;
   - SHA;
   - resultados de testes;
   - pendencias conhecidas.

6. Montar checklist de nao-regressao:
   - login;
   - credito;
   - objetivo;
   - T00;
   - T02;
   - aula;
   - A/B/C;
   - sinal 1/2/3;
   - feedback;
   - duvida;
   - drawer;
   - backup;
   - sync;
   - audio;
   - imagem.

Testes obrigatorios:

- todos os testes atuais;
- build release;
- instalacao manual do APK em Android real quando houver aparelho.

Criterio de aceite:

- main limpo;
- testes passam;
- build passa;
- APK baseline disponivel;
- documento baseline criado;
- nenhum codigo alterado sem necessidade.

Proibicoes:

- implementar feature 10x antes do baseline;
- refatorar;
- mexer em SimWeb;
- aceitar workspace sujo sem explicacao.

Saida esperada:

Equipe sabe exatamente qual e o ponto zero e como provar regressao.

======================================================================
FASE 1 - PROVA FISICA DO APK REAL E HARNESS DE REGRESSAO
======================================================================

Objetivo:

Eliminar o bloqueio recorrente: "passa em teste, mas ainda nao foi provado
no celular real".

Por que vem agora:

O SIM Flutter e um app Android. A superioridade sobre o Web so vale se o APK
real funciona na mao do aluno.

Orgaos envolvidos:

- QA;
- release APK;
- scripts de smoke test;
- docs;
- Flutter test driver, integration_test ou ferramenta equivalente.

Entregaveis:

1. `docs/10x-apk-real-test-protocol.md`.
2. `test/integration/` ou equivalente com fluxo vital automatizavel.
3. Relatorio de teste manual por aparelho.
4. Screenshots de fluxo vital.

Passos construtivos:

1. Definir aparelhos minimos:
   - Android pequeno;
   - Android medio;
   - Android com fonte grande;
   - Android lento;
   - Android recente.

2. Criar protocolo manual:
   - instalar APK;
   - abrir app;
   - login;
   - ver credito;
   - escolher idioma;
   - preencher objetivo;
   - anexar opcionalmente uma imagem;
   - rodar T00;
   - rodar T02;
   - abrir aula;
   - responder A/B/C;
   - escolher sinal;
   - ver feedback;
   - abrir duvida;
   - abrir drawer;
   - exportar/importar backup.

3. Criar harness automatizado gradualmente:
   - primeiro smoke test de abertura;
   - depois login fake/test account quando permitido;
   - depois fluxo de aula com backend controlado;
   - depois fluxo real contra API.

4. Criar pasta de evidencias:
   - screenshots;
   - logs;
   - SHA do APK;
   - device model;
   - Android version.

Testes obrigatorios:

- `flutter test`;
- build release;
- smoke manual em pelo menos 1 aparelho real.

Criterio de aceite:

- fluxo vital real passa em aparelho;
- se falhar, erro vira issue/fase de correcao;
- screenshots existem;
- protocolo repetivel.

Proibicoes:

- declarar B/10x sem APK real;
- usar apenas emulador Linux;
- esconder erro manual.

Saida esperada:

O time passa a ter uma porta real de produto. Toda fase futura precisa
passar por ela.

FASE 2 - EVENTO PEDAGOGICO CANONICO E TELEMETRIA LOCAL
======================================================================

Objetivo:

Criar a base de dados viva para medir aprendizagem: cada toque importante
vira evento pedagogico auditavel.

Por que vem antes de mastery:

Nao da para calcular dominio 10x se o app nao registra evidencia suficiente.

Orgaos envolvidos:

- StudentLearningState;
- StudentLearningStateService;
- StudentStateStore;
- LessonRuntimeEngine;
- LessonAnswerProgressController;
- Aux room services;
- media services;
- sync/backups.

Entregaveis:

1. Modelo `LearningEvent`.
2. Enum de eventos canonicos.
3. Serializacao no StudentLearningState.
4. Export/import/sync dos eventos.
5. Testes de roundtrip.
6. Documento `docs/10x-learning-event-contract.md`.

Passos construtivos:

1. Auditar eventos ja existentes.

2. Definir schema:
   - eventId;
   - type;
   - lessonLocalId;
   - marker;
   - layer;
   - skillId opcional;
   - timestamp;
   - payload tipado;
   - requestId opcional;
   - offline flag;
   - appVersion.

3. Criar tipos de evento:
   - LESSON_OPENED;
   - MATERIAL_READY;
   - QUESTION_VISIBLE;
   - ANSWER_SELECTED;
   - SIGNAL_SUBMITTED;
   - FEEDBACK_VISIBLE;
   - NEXT_ADVANCED;
   - DOUBT_OPENED;
   - DOUBT_SUBMITTED;
   - DOUBT_ANSWERED;
   - REVIEW_SCHEDULED;
   - REVIEW_COMPLETED;
   - RECOVERY_STARTED;
   - RECOVERY_COMPLETED;
   - IMAGE_OFFER_SHOWN;
   - IMAGE_ACCEPTED;
   - AUDIO_STARTED;
   - AUDIO_STOPPED;
   - ERROR_SHOWN;
   - RETRY_TAPPED.

4. Inserir escrita nos orgaos corretos:
   - resposta no controller de resposta;
   - sinal no motor/progress controller;
   - feedback na runtime;
   - duvida no controller de duvida;
   - midia no media service;
   - sync no sync service.

5. Nao escrever evento direto em widget, exceto comando de UI que chama
   metodo de orgao.

6. Garantir idempotencia:
   - eventId estavel;
   - replay nao duplica;
   - import de backup nao duplica.

Testes obrigatorios:

- event_log_roundtrip_test;
- event_log_no_duplicate_on_import_test;
- answer_signal_feedback_event_test;
- doubt_event_test;
- media_event_test;

Criterio de aceite:

- eventos aparecem no estado;
- export/import preserva;
- sync preserva;
- nenhum evento sensivel loga token/base64;
- testes passam.

Proibicoes:

- analytics direto em widget;
- evento sem schema;
- logar texto sensivel em claro sem decisao;
- quebrar backup.

Saida esperada:

O SIM passa a ter uma linha do tempo pedagogica real.

FASE 3 - SKILL GRAPH E MODELO DE DOMINIO POR EVIDENCIA
======================================================================

Objetivo:

Transformar curriculo linear em grafo de habilidades. O app deve saber
qual habilidade esta sendo ensinada e quais prerequisitos estao por tras.

Por que vem depois dos eventos:

Os eventos precisam apontar para skills. Sem skill graph, evidencia fica
presa em marker/layer e nao vira dominio transferivel.

Orgaos envolvidos:

- StudentLearningState;
- T00 adapter;
- T02 adapter;
- StudentExperienceEngine;
- StudentLearningStateService;
- backup/sync;
- API T00 se necessario.

Entregaveis:

1. Modelo `SkillNode`.
2. Campo `skillGraph` no estado.
3. Mapeamento `CurriculumItem.skillIds`.
4. Contrato T00 atualizado para skills, se necessario.
5. Migracao para estado antigo.
6. Testes.

Passos construtivos:

1. Criar modelo:
   - id;
   - title;
   - description;
   - prerequisites;
   - dependents;
   - cognitiveLevel;
   - area;
   - misconceptions;
   - evidenceRequired;
   - visualHints.

2. Adicionar ao estado sem quebrar estados antigos:
   - se skillGraph ausente, gerar skill simples por marker;
   - se CurriculumItem sem skillIds, mapear marker -> skillId.

3. Atualizar T00:
   - pedir skill graph;
   - validar shape;
   - retry se contrato invalido;
   - fallback seguro: skill por item, nao aula falsa.

4. Atualizar T02 payload:
   - enviar skillId atual;
   - enviar prerequisitos relevantes;
   - enviar misconceptions conhecidas.

5. Garantir backup/sync:
   - export inclui skillGraph;
   - import antigo funciona;
   - cloud merge preserva.

Testes obrigatorios:

- skill_graph_roundtrip_test;
- legacy_curriculum_generates_skill_graph_test;
- t00_skill_graph_contract_test;
- t02_receives_skill_context_test;
- backup_preserves_skill_graph_test.

Criterio de aceite:

- toda aula tem skillId;
- todo evento novo pode carregar skillId;
- estados antigos continuam abrindo;
- T00 invalido nao quebra app;
- build passa.

Proibicoes:

- depender de IA para inferir skill a cada tela;
- quebrar curriculos antigos;
- criar skillGraph fora do estado canonico.

Saida esperada:

O SIM sabe o que esta ensinando, nao apenas qual tela esta mostrando.

FASE 4 - MASTERYTRUTHENGINE 10X E DECISAO POR EVIDENCIA
======================================================================

Objetivo:

Fazer o app decidir dominio por evidencia acumulada, nao por uma resposta
isolada.

Por que vem depois do skill graph:

Dominio precisa ser calculado por habilidade, nao apenas por item/layer.

Orgaos envolvidos:

- MasteryTruthEngine;
- LearningDecisionEngine;
- StudentLessonExecutor;
- LessonAnswerProgressController;
- StudentLearningStateService.

Entregaveis:

1. Modelo `MasteryEvidence`.
2. Mastery status por skill.
3. Calculo de falsa confianca.
4. Calculo de esquecimento provavel.
5. Motor de decisao usando mastery.
6. Testes de cenarios.

Passos construtivos:

1. Definir MasteryEvidence:
   - skillId;
   - marker/layer;
   - correct;
   - answerLetter;
   - signal;
   - responseTimeMs;
   - doubtAsked;
   - reviewContext;
   - recoveryContext;
   - timestamp.

2. Capturar evidencia:
   - ao responder;
   - ao sinalizar;
   - ao concluir revisao;
   - ao concluir recuperacao;
   - ao pedir duvida, como sinal de incerteza.

3. Calcular status:
   - nao_visto;
   - iniciando;
   - aprendendo;
   - quase;
   - dominado_fraco;
   - dominado_forte;
   - esquecimento_provavel;
   - falsa_maestria;
   - precisa_revisao;
   - precisa_recuperacao.

4. Integrar com LearningDecisionEngine:
   - acertou + sinal 1 + historico bom -> avanca;
   - acertou + sinal 2 -> avanca e agenda revisao;
   - acertou + sinal 3 -> reforco ou revisao curta;
   - errou + sinal 1 -> falsa confianca;
   - errou repetido -> recuperacao;
   - prerequisito quebrado -> bloqueia conclusao com explicacao.

5. Garantir que UI nao decide.

Testes obrigatorios:

- mastery_evidence_capture_test;
- mastery_false_confidence_test;
- mastery_review_needed_test;
- mastery_recovery_needed_test;
- decision_engine_uses_mastery_test;
- ui_does_not_decide_mastery_test.

Criterio de aceite:

- engine decide com base em evidencias;
- nenhum widget contem regra pedagogica;
- revisao/recuperacao recebem razao clara;
- estado persiste mastery;
- testes passam.

Proibicoes:

- "um acerto = dominado";
- "sinal 1 sempre pula tudo";
- hardcode de pedagogia em tela.

Saida esperada:

O app passa a ter verdade pedagogica confiavel.

======================================================================
FASE 5 - REVISAO ESPACADA
======================================================================

Objetivo:

Fazer o app chamar o aluno para revisar no momento certo.

Por que vem depois do mastery:

Revisao espacada depende do grau de dominio e da estabilidade da memoria.

Orgaos envolvidos:

- novo SpacedReviewScheduler;
- StudentLearningState;
- ReviewRoomService;
- StudentAuxRoomService;
- drawer/menu;
- painel do aluno futuro.

Entregaveis:

1. `SpacedReviewScheduler`.
2. Fila de revisao persistida.
3. Regras de intervalo.
4. UI minima de pendencias.
5. Testes.

Passos construtivos:

1. Criar modelo ReviewDue:
   - skillId;
   - marker;
   - dueAt;
   - priority;
   - reason;
   - sourceEvidenceId;
   - completedAt.

2. Regras iniciais:
   - acerto sinal 1: revisao leve em 24h;
   - acerto sinal 2: revisao em 10-30min ou fim da aula;
   - acerto sinal 3: revisao imediata curta;
   - erro sinal 1: revisao + falsa confianca;
   - erro repetido: recuperacao.

3. Integrar com ReviewRoomService:
   - review queue vem de ReviewDue;
   - ao concluir, atualizar mastery;
   - reagendar se necessario.

4. UI:
   - mostrar pendencias sem bloquear tudo;
   - quando pendencia e critica, explicar por que.

Testes obrigatorios:

- spaced_review_schedule_test;
- review_due_persists_test;
- review_completion_updates_mastery_test;
- review_due_survives_backup_sync_test.

Criterio de aceite:

- revisao aparece por regra;
- revisao some ao concluir;
- nao duplica;
- nao bloqueia indevidamente;
- teste passa.

Proibicoes:

- notificar revisao aleatoria;
- sumir pendencia sem evento;
- usar data local sem controle de timezone em serializacao.

Saida esperada:

O SIM passa a combater esquecimento.

======================================================================
FASE 6 - RECUPERACAO POR PRE-REQUISITO
======================================================================

Objetivo:

Quando o aluno trava, o app deve recuperar a base que falta, nao repetir a
mesma pergunta.

Por que vem depois de skill/mastery/review:

Recuperacao precisa saber qual habilidade falhou e qual prerequisito e causa.

Orgaos envolvidos:

- RecoveryRoomService;
- StudentAuxRoomService;
- LearningDecisionEngine;
- SkillGraph;
- T02 aux caller;
- StudentLearningState.

Entregaveis:

1. `PrerequisiteResolver`.
2. Recovery plan.
3. Prompt/contrato de recuperacao.
4. Bloqueio/liberacao de conclusao por recuperacao.
5. Testes.

Passos construtivos:

1. Criar RecoveryNeed:
   - skillId;
   - prerequisiteSkillId;
   - reason;
   - evidenceIds;
   - blocksCompletion;
   - status.

2. Detectar:
   - erro repetido;
   - sinal 3;
   - falsa confianca;
   - prerequisito marcado fraco.

3. Resolver prerequisito:
   - primeiro pelo skill graph;
   - se ausente, por heuristica local;
   - nunca depender de IA sem validacao.

4. Gerar recuperacao:
   - microexplicacao;
   - exemplo resolvido;
   - pergunta curta;
   - feedback;
   - retorno para aula principal.

5. Estado:
   - salvar recoveryNeed;
   - salvar tentativa;
   - limpar ao concluir;
   - manter historico.

Testes obrigatorios:

- prerequisite_resolver_test;
- recovery_blocks_completion_test;
- recovery_completion_unblocks_test;
- recovery_returns_to_main_lesson_test;
- recovery_does_not_change_main_position_wrongly_test.

Criterio de aceite:

- recuperacao inicia por razao correta;
- nao bagunca item/layer principal;
- bloqueia conclusao somente quando necessario;
- limpa pendencia corretamente.

Proibicoes:

- recuperar sempre L1 sem diagnostico;
- apagar tentativa original;
- deixar aluno preso sem explicacao.

Saida esperada:

O aluno que erra nao fica no loop. O app reconstrói a base.

3. Modo foco.
4. Feedback acionavel.
5. Semantics em controles.

   - target question;
   - target selectedOption;
   - target signalRow;
